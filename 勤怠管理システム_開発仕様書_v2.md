# 勤怠管理システム 開発仕様書 v2

> v1からの変更点はレビューを踏まえて追記。新規・修正箇所には【v2追記】を付与。

## 1. 概要

### 目的

小規模〜中規模組織向けの勤怠管理システムを構築する。

本システムは Google アカウント認証を前提としたサーバレス構成を採用し、低コストかつ運用負荷の少ないシステムを目指す。

### 対象規模

* 数名〜50名程度
* PC・スマートフォンから利用
* Firebase Blaze Plan（従量課金）で運用【v2修正：Spark Planから変更。Cloud Functionsの外部通信制限を回避するため】

## 4.2 出勤打刻

### 概要

出勤ボタン押下時に出勤時刻を記録する。

### 処理方式【v2修正】

クライアントから Firestore へ直接書き込まず、Callable Function（`clockIn`）経由で記録する。
理由：打刻時刻の改ざん・なりすまし防止、および異常系バリデーションをサーバ側に一元化するため。

### 異常系バリデーション【v2追記】

* 同日に出勤打刻が既に存在する場合はエラーを返す（二重打刻防止）

### 保存値

* clockIn
* Firestore serverTimestamp()

---

## 4.3 退勤打刻

### 概要

退勤ボタン押下時に退勤時刻を記録する。

### 処理方式【v2修正】

クライアントから Firestore へ直接書き込まず、Callable Function（`clockOut`）経由で記録する。

### 異常系バリデーション【v2追記】

* 当日の出勤打刻（clockIn）が存在しない場合はエラーを返す（退勤のみの打刻防止）
* 既に退勤打刻が存在する場合はエラーを返す（二重打刻防止）

### 保存値

* clockOut
* Firestore serverTimestamp()

---

## 4.4 打刻修正申請

### 入力項目

* 対象日
* 修正前時刻
* 修正後時刻
* 修正理由

### フロー【v2修正：承認処理をCloud Functions化】

```text
ユーザー
 ↓
修正申請（attendance_requests作成）
 ↓
Pending
 ↓
管理者承認（Callable Function: approveAttendanceRequest）
 ↓
Firestoreトランザクションで
 attendance_requests更新 + attendance本体更新を同時実行
```

### 要件

* 修正履歴を保持する
* 承認者を記録する
* 承認・却下はCloud Functions経由で行い、申請ステータス変更とattendance本体更新の不整合が起きないようトランザクション内で処理する【v2追記】

| 値                  | 説明   |

| paid_leave         | 有給   |

| special_leave      | 特別休暇 |

```text

 ↓
```

## 4.9 月次締め

### 概要

勤怠確定処理を行う。

### 処理方式【v2修正】

Callable Function（`closeMonthlyAttendance`）経由で実行する。対象月の attendance を走査し、`monthly_closings` を作成すると同時に、以降の書き込みを Security Rules で拒否する。

### 要件

* 締め後は編集不可
* 管理者のみ解除可能
* 締め解除中（closed=false に戻している間）に行われた打刻修正・有給消化の承認は、解除した管理者が責任を持って確認する運用とする【v2追記】
* Security Rules 側で `monthly_closings.closed == true` の対象月の attendance への書き込みを拒否する条件を設ける【v2追記】

## 4.11 有給残数表示

### 概要

従業員は現在の有給残数を確認できる。

### 算出方法

```text
残数
=
有給付与合計
-
有給消化合計
```

### 要件

残数はDBに保存せず、表示時に計算する。

### 算出方式【v2追記】

v1時点ではクライアント側（React）で leave_grants / leave_usage を読み取って計算する。
将来的に半休・時間単位有給に対応する際は計算ロジックが複雑化するため、Callable Function（`getLeaveBalance`）に集約する方針とし、呼び出しインターフェースのみ先行して決めておく。

# 6. Firestore Security Rules 要件

## 基本方針【v2修正】

attendance（出勤・退勤打刻）、attendance_requestsの承認反映、leave_requestsの承認反映、leave_usage、monthly_closings、leave_grants への**書き込み**は、クライアントから直接行わせず、すべて Cloud Functions（Admin SDK経由）に限定する。
クライアントからの直接書き込みは `allow write: if false;` とし、読み取りのみ下記の通り許可する。
これにより admin ロールであっても Security Rules 上は「全データ自由書き込み」を許可せず、実際の更新は Cloud Functions 内のロール検証（`request.auth.token.role == 'admin'` 相当）を通じてのみ可能にする。

## staff

許可（読み取りのみ）

* 自分の users 参照
* 自分の attendance 参照
* 自分の attendance_requests 参照（作成のみクライアント許可、更新は不可【v2修正】）
* 自分の leave_requests 参照（作成のみクライアント許可、更新は不可【v2修正】）
* 自分の leave_grants 参照
* 自分の leave_usage 参照

禁止

* 他人のデータ参照
* attendance への直接書き込み（出退勤打刻含む）【v2追記】
* 承認処理
* 月次締め変更
* 有給付与

---

## admin

許可

* 全データ参照
* CSV出力（読み取りベース）
* 承認処理・月次締め管理・有給付与管理は Cloud Functions 経由でのみ実行可能【v2修正：直接書き込みは禁止】

禁止【v2追記】

* attendance・leave_usage・monthly_closings 等への Firestore 直接書き込み（Cloud Functionsを介さない更新）

---

# 6.5 Cloud Functions構成【v2追記】

承認・打刻・締め処理など複数コレクションにまたがる更新は、整合性確保のため Callable Function + Firestore トランザクションで実装する。

| 関数名 | 呼び出し元 | 処理内容 |
| --- | --- | --- |
| `clockIn` | staff | 当日clockIn重複チェック → attendance作成・更新 |
| `clockOut` | staff | clockIn存在チェック・clockOut重複チェック → attendance更新 |
| `approveAttendanceRequest` | admin | attendance_requests更新（status, approvedBy, approvedAt） + attendance本体更新をトランザクション実行 |
| `rejectAttendanceRequest` | admin | attendance_requests更新（status=rejected, approvedBy, approvedAt） |
| `approveLeaveRequest` | admin | leave_requests更新 + （paid_leaveの場合）leave_usage作成をトランザクション実行 |
| `rejectLeaveRequest` | admin | leave_requests更新（status=rejected） |
| `grantLeave` | admin | leave_grants作成 |
| `getLeaveBalance` | staff/admin | leave_grants・leave_usageを集計し残数を返す（将来の半休・時間単位対応の窓口）|
| `closeMonthlyAttendance` | admin | 対象月のattendanceを確定し monthly_closings作成、以降の書き込みをRules側で拒否 |
| `reopenMonthlyAttendance` | admin | monthly_closings.closed を false に戻す |
| `exportAttendanceCsv` | admin | CSVデータ生成（Shift-JIS変換、JST変換込み）|

### 共通要件

* 全関数で `context.auth` を検証し、role（admin/staff）に応じたアクセス制御を行う
* 監査要件（7章）の createdBy / updatedBy / approvedBy 等は、Cloud Functions内で `context.auth.uid` を用いて自動付与する（クライアントからの指定値は信用しない）
